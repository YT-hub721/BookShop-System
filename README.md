## 2. 项目概述

   本项目旨在构建一套前后端分离的书店购物系统，解决传统书店购书场景中信息更新不及时、用户购物流程繁琐、数据管理不规范等问题。系统通过整合前端Vue框架与后端Spring Boot框架，实现用户端便捷购书与管理员端高效管理的双向功能，同时基于AJAX异步请求与JSON数据传输，优化页面交互体验，保障数据操作的安全性与高效性，最终打造一个功能完整、体验流畅、权限清晰的书店购物平台。
## 3. 核心目标用户

| 用户角色 | 描述 |
| :--- | :--- |
| 购书用户 | 读者群体，浏览、搜索图书，加入购物车，下单购买，查看个人订单历史 |
| 商城系统管理员 | 书店运营管理人员，负责图书上下架、库存管理、订单处理、用户管理等 |

## 4. 核心系统功能

### 4.1 购书用户功能

- **图书查询**：支持按书名等条件进行精确或模糊搜索，支持排序和筛选。
- **加入购物车**：用户在图书详情页选择数量后加入购物车，可随时修改数量或删除商品。
- **下单**：从购物车生成订单，填写收货地址、选择支付方式（模拟支付），下单后扣减库存。
- **查看个人订单**：按订单状态（待付款、待发货、已发货、已完成、已取消）分类展示，支持查看订单详情。

### 4.2 管理员功能

- **后台数据管理**：管理图书信息（增删改查、上下架）、查看销售统计。
- **订单处理**：查看所有用户订单，修改订单状态（待发货 → 已发货，填写物流单号），处理异常订单。

### 4.3 数据权限控制

- 普通用户登录后，只能查询、修改、删除属于自己的购物车和订单数据，无法访问其他用户的信息。
- 管理员拥有全平台数据读写权限，可查看和管理所有用户的订单、所有图书信息。

### 4.4 前后端分离交互方式

- 所有动态请求均采用 **AJAX** 异步发送，不刷新整个页面。
- 前后端数据交换使用 **JSON** 格式，接口规范统一。
- 前端根据后端响应结果 **局部刷新** 页面区域（如购物车数量、订单列表），提升用户体验。

## 6. 核心功能详细设计

### 6.1 加入购物车功能

**业务流程**：
1. 用户在图书详情页点击“加入购物车”按钮，前端获取图书ID和数量。
2. 前端发送 AJAX POST 请求到 `/api/cart/add`，请求体为 JSON：`{"bookId": 123, "quantity": 1}`，请求头携带 JWT Token。
3. 后端验证 Token，确认用户身份；检查图书库存是否充足；将购物车记录存入数据库（或更新已有记录数量）。
4. 后端返回 JSON 响应：`{"code": 200, "message": "添加成功", "data": {"cartTotalCount": 3}}`。
5. 前端根据返回结果，局部更新页面右上角购物车图标旁的徽章数字，并弹出提示“添加成功”。

**涉及数据表**：`cart`（购物车表），主要字段：`cart_id`、`user_id`、`book_id`、`quantity`、`create_time`、`update_time`。

### 6.2 管理员订单处理功能

**业务流程**：
1. 管理员登录后台，进入“订单管理”页面，前端通过 AJAX 请求 `/api/admin/orders?status=待发货` 获取所有待发货订单列表。
2. 页面以表格形式展示订单，每条订单后带有“确认发货”按钮。
3. 管理员点击“确认发货”，弹出对话框填写物流单号。
4. 前端发送 PUT 请求到 `/api/admin/orders/{orderId}/ship`，请求体 JSON：`{"trackingNumber": "SF1234567890"}`。
5. 后端验证管理员权限，更新订单状态为“已发货”，保存物流单号，并记录发货时间。
6. 后端返回成功响应，前端局部刷新订单列表，将该订单移出“待发货”列表或更新状态显示。

**涉及数据表**：`orders`（订单表），主要字段：`order_id`、`user_id`、`total_amount`、`status`、`tracking_number`、`create_time`、`ship_time`。
